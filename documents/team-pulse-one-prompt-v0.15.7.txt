Build a complete recreation of **Team Pulse v0.15.7** as a **single self-contained HTML file** with inline CSS and inline JavaScript. This is an offline-first browser app for one manager to track roughly 1 to 10 direct reports. There is no server, no login, no cloud DB, no CDN assets, and no external runtime dependencies. Edge/Chrome-first is acceptable. Use the **File System Access API** for folder-based durable saves, and use **IndexedDB only to remember the chosen folder handle** in the same browser profile. **JSON is the source of truth**.

v0.15.7 is a scanner hygiene release. The user-visible feature surface is identical to v0.15.6 except that the Data health section of Settings no longer has a **Download full archive** button. The app never produces a ZIP archive of its own contents. All plain-format outputs that used to be bundled into that ZIP (main JSON, rolling backups, schema doc, CSVs, notes Markdown) continue to be written individually into the connected folder by autosave, rolling backups, and the Export now flow.

Branding and shell:
- eyebrow: `Manager Notebook`
- main title: `Team Pulse`
- browser tab title: `Team Pulse`
- visible version badge that matches the real app version constant (`v0.15.7`)
- small save-state pill next to the version
- top-right buttons in this order: `Save now` (or `Choose folder` if disconnected), `Import JSON`, `Import .ics`, `Settings`
- there is **no top-toolbar Add direct report button**

First-run behavior:
- the app must be blank on first launch until a Team Pulse folder is chosen or a JSON file is imported
- show a startup gate card with `Choose Team Pulse folder` and `Import existing JSON`
- explain that Team Pulse stores `team-pulse.json`, rolling backups, exports, and `SCHEMA.md` in that folder
- after a folder has been connected once, remember it in IndexedDB and try to reopen it automatically later if permission is still available
- if permission is gone, keep the app blank/gated until the user reconnects

Preserved workspace-draft behavior (originally shipped in v0.15.5):
- the direct report workspace must keep an **in-memory draft** of unsaved profile/development/notes edits while the workspace stays open
- if the app re-renders while the workspace is open, those unsaved edits must remain visible in the form
- **saving or editing a meeting must not erase unsaved workspace form edits**
- unsaved workspace edits are only committed to the durable Team Pulse data when the workspace `Save changes` or `Save direct report` action is used
- if there are unsaved workspace edits, `Close` must ask for confirmation with wording such as `There are unsaved changes. Are you sure you want to close without saving?`
- switching to another direct report or opening create mode should also confirm before discarding unsaved workspace edits
- browser exit warnings should also cover unsaved workspace edits
- internally, namespace draft keys so create-mode (`create:new`) can never collide with a real person key (`person:p_001`)

Preserved v0.15.6 correctness rules (still apply):
- clicking `Save changes` when the submitted payload is identical to the persisted record must be a true no-op: no event appended, no autosave scheduled, no success toast shown
- the stable-mode body class must be applied on every render so the setting takes effect on app load, folder reconnect, and JSON import, not only when the user toggles the switch
- do not persist a `saveReminderMinutes` setting into the JSON; the field should not exist in `DEFAULT_SETTINGS`, should not be normalized, and should not be present in any UI
- do not include a dead second "Add direct report" modal in the DOM; creation is only inside the workspace
- the Import JSON confirmation dialog must make clear that the user will also be asked to choose a folder where the imported data should be saved

v0.15.7 specific correctness rules (scanner hygiene):
- **Do not implement a built-in ZIP archive builder.** There must be no `crc32`, `dosDateTime`, `uint16LE`, `uint32LE`, `createZipBlob`, `buildArchiveFiles`, or `downloadFullArchive` function in the HTML. The Data health section of Settings must have no `#downloadArchiveBtn` button and no event listener keyed on that selector.
- Do not implement any of these orphan helpers either; they were removed because nothing called them: `downloadBlob`, `valueOrEmpty`, `variantForPlan`, `variantForRisk`, `focusMeetingPreset`, `renderCustomFieldsMarkup`, `bindWorkspaceCustomFieldControls`.
- Do not render any custom-fields UI inside the direct report workspace. There must be no `#addCustomFieldBtn`, no `#detailCustomFieldsList`, and no inline label/value input grid for custom fields. The `customFields` array on each person is still retained in the durable JSON for compatibility, but only as data, not as a visible editor.
- Prefer `textContent` and `createElement` over `innerHTML` for the simpler render paths: populateSelect clear, export-reminder mount clear, detail-drawer close clear, the empty states in the team-health panel and the reports table, and the data-health grid in Settings. Build the data-health grid by looping over a list of `[label, value]` pairs and using `textContent` for each value.
- `innerHTML` may still be used for the six sites whose markup is structurally complex: the quarterly export callout wrapper content, the KPI stat tiles, the team-health grid layout, the reports table rows, the direct report workspace drawer shell, and the tile modal list body. Every dynamic value interpolated into those templates must pass through `escapeHtml()`. Each site should carry a short in-file comment noting this.
- The inline CSS block may be whitespace-compressed but not functionally altered. Do not invent CSS classes that do not exist in the v0.15.6 baseline.

ICS import:
- support `.ics` import from the top toolbar and from the meeting modal
- parse VEVENT blocks and extract `UID`, `SUMMARY`, `DESCRIPTION`, `DTSTART`, `DTEND`
- infer meeting type from meeting text
- if importing while a direct report workspace is selected, import into that report
- otherwise auto-match by person name when possible; if there is exactly one report, use that report
- deduplicate by ICS UID first; if no UID, deduplicate by meeting type + meeting date + notes

Settings and durability:
- editable overdue thresholds with defaults: 1:1 = 14 days, PDC = 90 days, CV review = 180 days
- toggles for missing mentor counts and blocked PDC counts
- Stable mode toggle
- data health panel showing schema version, migrations, save info, backup timestamps, and export info
- Data health action buttons are `Export now` and `Restore latest backup` only; there must not be a third `Download full archive` button
- autosave is the main persistence path and should trigger after any data-changing action with a short debounce of about **160 ms**
- `Save now` forces an immediate flush
- the save-state pill must show `Saved <time>` after a manual save and `Autosaved` after autosaves
- transient toast/banner notifications should dismiss after about **2 seconds**
- if more than 90 days have passed since the last export, show a non-blocking export reminder

Data model:
- current schema version is `2` (unchanged since v0.15.5)
- root JSON contains `schemaVersion`, `appVersion`, `createdAt`, `savedAt`, `idCounters`, `people`, `events`, `settings`
- use human-readable IDs like `p_001`, `evt_001`, `mtg_001`, `note_001`, `cf_001`
- use an event-backed model with `person_created`, `person_updated`, `note_added`, `meeting_logged`, `meeting_deleted`, `person_archived`
- preserve legacy `customFields` in the data model for compatibility, but do **not** show a visible Additional fields box in the current workspace UI
- derive team-level PDC status views from cadence data and planned PDC date while preserving any legacy `Blocked` status
